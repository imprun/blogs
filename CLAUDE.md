# 블로그 작성 가이드

> **AI Agent 필독**: 이 문서는 imprun.dev 블로그 작성 시 **반드시 따라야 할 강제 규칙**입니다.

## 핵심 원칙

**imprun.dev 기술 블로그는 전문 기업 기술 블로그 스타일을 따릅니다.**

- **회사 기술 블로그 스타일**: Netflix Tech Blog, Uber Engineering 수준의 전문적인 톤
- **객관적이고 기술적**: 감정적 표현이나 과장 없이 사실 중심
- **이모지 사용 자제**: 기본적으로 사용하지 않음
- **AI 티 제거**: "오늘은", "~해볼까요?", "~해봅시다" 등 금지
- **정확성 우선**: 추정치는 명확히 표기

**규칙 위반 시 즉시 재작성 요청됨.**

## 기술 설명 원칙: "번역이 아닌 설명"

> **"어려운 것을 어렵게 설명하는 건 쉽습니다. 어려운 것을 쉽게 설명하는 것이 진짜 실력입니다."**

기술 블로그의 목적은 **정확한 정보 전달**이 아니라 **이해를 돕는 것**이다.

### 설명의 3가지 원칙

**1. 상대방의 배경에서 출발하라**

독자가 **이미 알고 있는 개념**에서 시작한다.

```markdown
❌ 나쁜 예시
"gRPC는 HTTP/2 기반의 바이너리 직렬화 프로토콜입니다."

✅ 좋은 예시
"REST API가 편지라면, gRPC는 전화입니다.
- 편지(REST): 매번 새 봉투, 텍스트로 작성, 일방향
- 전화(gRPC): 연결 유지, 즉시 대화, 양방향"
```

**2. 구체적 이미지를 활용하라**

추상적 개념을 **눈에 보이는 사물**로 전환한다.

| 추상적 개념 | 구체적 비유 |
|------------|-----------|
| API Gateway | 호텔 컨시어지 |
| 멀티테넌시 | 아파트 동/호 |
| RBAC | 회사 출입증 등급 |
| Reconciliation | 체온 조절하는 에어컨 |
| 스냅샷 | 게임 세이브 파일 |

**3. "왜"를 먼저 설명하라**

기술보다 **문제**를 먼저 제시한다.

```markdown
❌ 나쁜 예시
## Oathkeeper란?
Oathkeeper는 Identity & Access Proxy입니다...

✅ 좋은 예시
## 왜 모든 API에 인증 코드를 넣으면 안 될까?
매번 토큰 검증 로직을 복붙하면:
- 실수로 빠뜨리면 보안 구멍
- 변경 시 모든 서비스 수정 필요
- 테스트도 중복

→ 그래서 Oathkeeper가 **한 곳에서** 처리한다.
```

### 용어 설명 패턴

**패턴 1: 비유 + 정의**
```markdown
**API Gateway**는 호텔의 컨시어지와 같다.
모든 손님(요청)이 컨시어지(Gateway)를 통해야 객실(서비스)에 도착한다.
```

**패턴 2: 문제 → 해결책**
```markdown
API를 만들 때마다 인증 코드를 복붙하시나요?
**Oathkeeper**가 한 곳에서 모든 인증을 처리한다.
```

**패턴 3: Before/After**

| 이전 | 이후 |
|------|------|
| kubectl로 YAML 수동 배포 | 웹 콘솔에서 클릭 한 번 |
| 인증 로직 직접 구현 | Ory 스택 자동 연동 |

**패턴 4: Q&A 형식**
```markdown
> **Q: Namespace가 수백 개 생기면 성능 문제가 생기지 않나요?**
>
> 아니요. Kubernetes는 수천 개의 Namespace를 지원한다.
> Namespace는 논리적 그룹일 뿐, 물리적 리소스를 소비하지 않는다.
```

### 피해야 할 것

**1. 지식의 저주**
```markdown
❌ "Reconciliation Loop는 선언적 상태 관리의 핵심입니다."
   → 전문 용어로 전문 용어를 설명

✅ "에어컨 온도 조절기를 생각해보세요.
   '25도로 맞춰줘'라고 설정하면, 에어컨이 알아서 현재 온도를 확인하고
   25도가 될 때까지 냉방/난방을 합니다.
   Kubernetes도 똑같이 '원하는 상태'만 선언하면 알아서 맞춰줍니다."
```

**2. 완벽한 정확함 추구**
```markdown
❌ 100% 정확하지만 이해 불가능한 설명
✅ 70% 정확하지만 이해 가능한 설명

예: "HTTP/2는 여러 요청을 동시에 보낼 수 있다" (충분)
    vs "HTTP/2 멀티플렉싱은 HPACK 압축과 스트림 기반..." (과잉)
```

**3. 예시 없는 개념 나열**
```markdown
❌ "Oathkeeper의 파이프라인: Authenticator → Authorizer → Mutator"

✅ "Oathkeeper의 파이프라인은 공항 보안 검색대와 같다:
   1. Authenticator: '여권 보여주세요' (신원 확인)
   2. Authorizer: '이 비행기 탑승권 맞나요?' (권한 확인)
   3. Mutator: '탑승권에 게이트 번호 스탬프' (정보 추가)"
```

## 필수 블로그 구조

1. **메타데이터**: 작성일, 카테고리, 키워드
2. **요약**: 3-4문장 핵심 요약
3. **문제 상황**: 증상, 환경 구성, 초기 진단
4. **근본 원인 분석**: 기술 메커니즘, 설정 오류
5. **해결 과정**: 단계별 명령어와 검증
6. **재발 방지 방안 또는 교훈**
7. **참고 자료**: 관련 문서, 공식 문서 링크
